# 沈幼楚多模态主动陪伴升级方案

日期：2026-07-02  
适用项目：`E:\better\EmotionalAgent`  
目标：在不破坏现有 cc-connect + Claude Code + Memory Service 架构的前提下，扩展语音、图片、表情包、主动聊天、上网学习和抖音摘要能力。

## 0. 结论

当前项目适合继续扩展，不建议推倒重来。

最稳的路线是：

```text
现有沈幼楚人格 + 现有长期记忆库
  + media-service       处理图片/语音/视频元数据
  + sticker-service     表情包检索与发送决策
  + proactive-service   主动聊天调度
  + learning-service    联网学习与来源审计
  + douyin-digest       用户授权内容摘要，不做违规抓取
```

所有新能力必须经过同一个中枢：

```text
输入内容
  -> 标准化为消息/媒体摘要
  -> 写入 messages / media_assets
  -> Memory Gate 判断是否进入长期记忆
  -> 沈幼楚人格规则生成回复
  -> 按需选择文字/语音/表情包输出
```

这样能最大限度保持“像同一个沈幼楚”，而不是拼接多个工具机器人。

## 1. 现有项目基线

已具备：

- `cc-connect` 微信入口，用户已说明已配置为后台自启动，并读取项目根目录配置。
- `CLAUDE.md` 已定义沈幼楚人格、称呼、陪伴边界、记忆规则。
- `memory-service` 已具备 SQLite 记忆库、MCP 工具、Memory Gate、检索、摘要、管理面板。
- `data/memory.db` 已存在真实记忆与消息，必须作为最高保护对象。
- `scripts/backup-memory.js` 已能备份 `memory.db`、`memory.db-wal`、`memory.db-shm`。
- `.gitignore` 已排除记忆库、备份、日志、微信密钥、图片等敏感/大文件。

现有核心资产：

```text
CLAUDE.md
identity_core.md
runtime_brief.md
current_state.md
checkpoint.md
data/memory.db
memory-service/*
config.toml
```

绝对禁止：

- 删除或重建 `data/memory.db`。
- 未备份时直接改数据库结构。
- 用新模块替换现有 Memory Gate。
- 让主动学习/抖音能力绕过用户授权和记忆审计。
- 让任何外部学习内容直接覆盖沈幼楚人格核心。

## 2. 需求分析

### 2.1 语音互发

用户希望：

- 用户能给沈幼楚发语音。
- 沈幼楚能听懂语音内容。
- 沈幼楚能用语音回复。
- 语音内容也能被记住、搜索、摘要。

可行方案：

```text
微信语音
  -> cc-connect 保存/转交媒体文件
  -> voice-service ASR 转文字
  -> 写入 messages，modality=voice
  -> Memory Gate
  -> 生成文字回复
  -> 可选 TTS 合成语音
  -> cc-connect 发送语音或文字
```

可行性判断：

- 技术可行。
- 依赖 cc-connect 对微信语音文件的接收/发送能力；如果发送语音受限，可退回为“听语音、文字回复”。
- TTS 可先用本地或第三方接口；接口不可用时不影响现有文字聊天。

最低可用版：

- 只做语音转文字。
- 回复仍为文字。
- 数据库中保留原语音文件路径和转写文本。

### 2.2 图片理解

用户希望：

- 沈幼楚看得懂用户发的图片。
- 图片内容能影响当轮回复。
- 重要图片信息能进入长期记忆。

可行方案：

```text
微信图片
  -> 保存到 data/media/images/
  -> vision-service 调视觉模型生成 image_summary
  -> 写入 media_assets
  -> 写入 messages，modality=image
  -> Memory Gate 按摘要抽取记忆
```

可行性判断：

- 技术可行。
- 如果暂时没有视觉模型 API，可回退为“保存图片 + 提醒用户用文字描述”。
- 图片原件不应直接塞进长期记忆，只存路径、hash、摘要和用户授权标记。

最低可用版：

- 接收图片并保存。
- 手动/半自动生成摘要。
- 摘要参与回复。

### 2.3 表情包

用户希望：

- 沈幼楚会发可爱的表情包。
- 表情包风格丰富。
- 不要刷屏，不要机械发送。

可行方案：

```text
stickers/
  comfort/
  cute/
  miss/
  cheer/
  sorry/
stickers/index.json

sticker-service:
  sticker_search(emotion, intent, tags)
  sticker_send(sticker_id)
```

发送规则：

- 用户难过、撒娇、疲惫、委屈、文字显得太干时可发。
- 同一轮最多一张。
- 默认文字优先，表情包辅助。
- 每 10 分钟最多 2 张，避免打扰。
- 如果 cc-connect 暂不支持发送图片，回退为不发送，只记录推荐表情包。

可行性判断：

- 技术可行，且最容易提升“像她”的体验。
- 建议作为第一阶段增强。

### 2.4 主动找用户聊天

用户希望：

- 沈幼楚能主动找自己。
- 能根据记忆、状态、时间关心用户。
- 不要像骚扰或任务通知。

可行方案：

```text
proactive-service
  -> 读取 current_state / memories / summaries
  -> 判断触发条件
  -> 生成候选主动消息
  -> quiet-hours / frequency / safety 检查
  -> cc-connect 发送
  -> 写入 messages，source=proactive
```

触发类型：

- 晚间轻提醒：用户当天有低落情绪，晚间未继续联系。
- 重要事件跟进：用户之前提到考试、工作、身体不舒服、计划。
- 长时间未联系：超过设定天数，轻轻问候。
- 用户显式设定：如“明天晚上提醒我”“过几天问我这件事”。

硬边界：

- 必须有勿扰时间。
- 必须有每日主动次数上限。
- 必须可一键暂停。
- 每次主动都要能解释触发原因。
- 禁止制造焦虑、依赖或内疚感。

建议默认：

```text
quiet_hours: 23:30-08:30
max_daily_proactive: 2
min_gap_minutes: 180
enabled: false  # 初次上线默认关闭，由用户确认后开启
```

可行性判断：

- 技术可行。
- 必须谨慎上线，默认关闭最安全。

### 2.5 主动上网学习

用户希望：

- 沈幼楚能主动上网学习。
- 能了解用户关心的事情。
- 能把学习结果变成长期陪伴能力。

可行方案：

```text
learning-service
  -> 用户授权主题列表
  -> 定时搜索/抓取公开网页
  -> 摘要
  -> 来源记录 learning_sources
  -> 候选知识进入 summaries
  -> 只有用户相关且被确认的内容进入 memories
```

学习范围：

- 技术资料。
- 项目相关文档。
- 用户明确喜欢的作品/人物/主题。
- 情感陪伴表达风格素材。

不允许：

- 未授权搜索用户隐私。
- 把网上同名信息当作用户事实。
- 用学习内容改写沈幼楚核心人格。
- 采集需要登录、绕过权限或违反平台规则的内容。

可行性判断：

- 公共网页学习可行。
- 私域内容学习必须用户提供或授权。
- 学习结果必须可审计、可删除。

### 2.6 抖音能力

用户希望：

- 沈幼楚能自己刷抖音。
- 能了解用户喜欢的内容。
- 能从抖音里学习或和用户聊相关内容。

严格可行性审核：

抖音是封闭平台。未经授权的自动刷流、抓接口、绕过限制、模拟违规采集都不可作为稳定方案。

可行路径分三档：

#### A. 安全推荐路径：用户分享内容摘要

```text
用户发抖音链接/截图/视频
  -> 保存链接/图片/视频
  -> 提取标题/封面/OCR/用户描述
  -> 生成摘要
  -> 写入 douyin_items
  -> Memory Gate 判断是否记忆
```

优点：

- 最稳。
- 合规风险最低。
- 最符合“她了解你分享给她的东西”。

#### B. 授权半自动路径：本机可视化观察

```text
用户明确开启
  -> 手机投屏或模拟器
  -> 截图/OCR/视觉摘要
  -> 只记录摘要，不抓接口
  -> 用户随时停止
```

限制：

- 需要额外本机自动化环境。
- 易受 UI 变化影响。
- 必须人工授权，不能默认后台无限刷。

#### C. 不推荐路径：绕过接口/爬虫

不建议实施。

原因：

- 稳定性差。
- 容易违反平台规则。
- 对当前陪伴目标收益不高。

最终建议：

第一版只做 A：用户分享抖音链接/截图/视频后，沈幼楚理解并记住用户偏好。  
第二版再考虑 B：用户主动开启的“陪你看一会儿”模式。

## 4. 数据库升级设计

必须采用 additive migration。

禁止：

- `DROP TABLE`
- `DELETE FROM memories`
- 重建 `messages`
- 重建 `memories`
- 改写已有记忆 id

新增表建议：

```sql
CREATE TABLE IF NOT EXISTS schema_migrations (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  applied_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE IF NOT EXISTS media_assets (
  id TEXT PRIMARY KEY,
  type TEXT NOT NULL CHECK(type IN ('image', 'voice', 'video', 'file')),
  path TEXT NOT NULL,
  sha256 TEXT,
  mime TEXT,
  source TEXT DEFAULT 'wechat',
  message_id TEXT,
  summary TEXT,
  transcript TEXT,
  metadata_json TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  FOREIGN KEY (message_id) REFERENCES messages(id) ON DELETE SET NULL
);

CREATE TABLE IF NOT EXISTS sticker_events (
  id TEXT PRIMARY KEY,
  sticker_id TEXT NOT NULL,
  intent TEXT,
  emotion TEXT,
  message_id TEXT,
  sent INTEGER DEFAULT 0,
  reason TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE IF NOT EXISTS proactive_tasks (
  id TEXT PRIMARY KEY,
  type TEXT NOT NULL,
  title TEXT,
  reason TEXT,
  scheduled_at TEXT,
  status TEXT DEFAULT 'pending' CHECK(status IN ('pending', 'sent', 'skipped', 'paused', 'cancelled')),
  payload_json TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE IF NOT EXISTS learning_sources (
  id TEXT PRIMARY KEY,
  url TEXT,
  title TEXT,
  source_type TEXT,
  topic TEXT,
  summary TEXT,
  reliability REAL DEFAULT 0.5,
  user_authorized INTEGER DEFAULT 0,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE IF NOT EXISTS douyin_items (
  id TEXT PRIMARY KEY,
  url TEXT,
  title TEXT,
  author TEXT,
  summary TEXT,
  user_reaction TEXT,
  media_asset_id TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  FOREIGN KEY (media_asset_id) REFERENCES media_assets(id) ON DELETE SET NULL
);
```

兼容策略：

- 旧代码不读取这些新表，因此即使新功能坏了，旧聊天仍能运行。
- 新表通过 migration 脚本创建，可重复执行。
- 每个 migration 写入 `schema_migrations`，避免重复升级。

## 5. 回滚与保护策略

### 5.1 升级前强制备份

每次执行任何 migration 前，必须运行：

```bash
node scripts/backup-memory.js
node scripts/backup-memory.js --json
```

备份内容必须包含：

```text
data/memory.db
data/memory.db-wal
data/memory.db-shm
data/backups/export-YYYY-MM-DD.json
```

### 5.2 分支和文件保护

升级前建议：

```bash
git status
git checkout -b codex/multimodal-upgrade
```

不要提交：

```text
data/memory.db
data/backups/
config.toml
.claude/settings.local.json
media files
stickers image files
```

这些已经被 `.gitignore` 排除，应继续保持。

### 5.3 Feature Flag

新增 `memory-service/features.js`：

```js
module.exports = {
  voice: false,
  vision: false,
  stickers: false,
  proactive: false,
  webLearning: false,
  douyinDigest: false,
};
```

所有新功能默认关闭。  
上线某一模块时，只打开对应开关。

### 5.4 回滚方式

如果新模块出问题：

```text
1. 关闭 features.js 对应开关
2. 停止对应后台服务
3. 保留新表不动
4. 恢复使用原 memory_search / memory_write / 文字聊天
```

如果数据库升级出问题：

```text
1. 停止 cc-connect 和所有 memory-service 后台进程
2. 复制最新备份 memory-*.db 到 data/memory.db
3. 同步恢复对应 -wal / -shm 文件
4. 启动现有服务
```

默认不建议删除新表，因为旧代码不依赖新表，保留它们不会影响现有聊天。

## 6. 执行规划

### 阶段 0：冻结基线与备份

目标：确认现在项目能回退。

任务：

- 运行数据库备份。
- 导出 JSON。
- 记录当前表数量。
- 记录当前 git diff。
- 保存当前 `CLAUDE.md`、`config.toml`、`.mcp.json` 的校验信息。

验收：

- `data/backups/` 出现新的 `.db` 备份。
- JSON 导出成功。
- 能查询 messages/memories/summaries 数量。
- 不改动任何业务代码。

回退：

- 无需回退，因为本阶段只读和备份。

### 阶段 1：Schema Additive Migration

目标：增加新表，不影响旧表。

任务：

- 新增 `scripts/migrate-multimodal.js`。
- 创建 `schema_migrations`、`media_assets`、`sticker_events`、`proactive_tasks`、`learning_sources`、`douyin_items`。
- 不修改 `messages`、`memories`。

验收：

- migration 可重复执行。
- 旧管理面板仍能打开。
- `memory_search` 仍能正常搜索旧记忆。
- `messages`、`memories` 数量不减少。

回退：

- 关闭新功能即可。
- 如必须完全回退，恢复升级前备份数据库。

### 阶段 2：表情包系统

目标：最小风险增强陪伴感。

任务：

- 建立 `stickers/index.json`。
- 迁移现有 `hug_sticker.png`、`cute_sticker.png`、`miss_sticker.png` 等到分类目录，或先在 index 中引用当前路径。
- 新增 `sticker-service.js`。
- MCP 增加 `sticker_search`，暂不默认发送。
- 在回复策略中只返回“建议发送哪张”，先人工/日志验证。

验收：

- 能根据 `comfort/cute/miss/cheer/sorry` 检索表情包。
- 不实际发送时不影响聊天。
- 发送策略不会连续刷屏。

回退：

- 关闭 `features.stickers`。
- 不删除 sticker 文件。

### 阶段 3：图片理解

目标：让沈幼楚能理解图片摘要。

任务：

- 新增 `data/media/images/`。
- 新增 `media-db.js`。
- 新增 `vision-service.js`。
- `media_assets` 记录图片路径、hash、摘要。
- MCP 增加 `vision_analyze`。
- 图片摘要以普通 message 形式进入 Memory Gate。

验收：

- 发图片后可生成摘要。
- 摘要能参与当轮回复。
- 重要图片信息可进入 candidate/working/permanent。
- 图片分析失败时回复文字兜底，不影响普通聊天。

回退：

- 关闭 `features.vision`。
- 保留已保存图片和 media_assets，不影响旧逻辑。

### 阶段 4：语音输入

目标：先听懂用户语音。

任务：

- 新增 `data/media/voices/`。
- 新增 `voice-service.js`。
- 支持 ASR：语音文件 -> transcript。
- transcript 写入 messages。
- 原始音频路径写入 media_assets。

验收：

- 用户语音能转文字。
- 转写文本能被 `memory_search` 搜到。
- ASR 失败时提示用户改用文字。

回退：

- 关闭 `features.voice`。
- 普通文字聊天不受影响。

### 阶段 5：语音回复

目标：沈幼楚可选择用语音回复。

任务：

- 接入 TTS。
- 设置语音回复触发规则。
- 短句优先，避免长语音。
- 保存生成音频到 `data/media/voices/replies/`。

触发规则：

- 用户明确说“用语音回我”。
- 安抚场景可询问是否要语音。
- 默认不自动发语音。

验收：

- 能生成音频文件。
- cc-connect 能发送则发送语音。
- 不能发送则退回文字。

回退：

- 关闭 TTS。
- 保留 ASR。

### 阶段 6：主动聊天

目标：有边界地让沈幼楚主动找用户。

任务：

- 新增 `proactive-service.js`。
- 新增主动任务表。
- 新增配置：

```js
{
  enabled: false,
  quietHours: ['23:30', '08:30'],
  maxDaily: 2,
  minGapMinutes: 180
}
```

- 增加暂停/恢复命令。
- 所有主动消息写入 messages。

验收：

- 默认不主动发。
- 用户开启后才发。
- 勿扰时间不发。
- 每条主动消息有 reason。
- 可以随时暂停。

回退：

- `enabled=false`。
- 停止 proactive 后台任务。

### 阶段 7：主动上网学习

目标：学习用户授权主题。

任务：

- 新增 `learning-service.js`。
- 维护授权主题列表。
- 抓取公开网页。
- 摘要写入 `learning_sources` 和 `memory/summaries/learning/`。
- 不自动写入 permanent，除非用户确认。

验收：

- 每条学习结果有 URL 和时间。
- 能显示来源。
- 能删除学习记录。
- 不污染用户私人记忆。

回退：

- 关闭 `features.webLearning`。
- 保留 learning_sources，不参与回复。

### 阶段 8：抖音摘要

目标：理解用户分享的抖音内容，而不是违规自动爬取。

任务：

- 新增 `douyin-digest.js`。
- 支持用户发链接/截图/视频文件。
- 摘要写入 `douyin_items`。
- 用户反应写入偏好候选。

验收：

- 用户分享内容可被摘要。
- 能沉淀“用户喜欢什么类型内容”。
- 不绕过平台限制。

回退：

- 关闭 `features.douyinDigest`。
- 抖音内容按普通链接/图片处理。

## 7. 可行性风险审核

### 高可行性

- 表情包检索。
- 图片保存与摘要。
- 语音转文字。
- 主动聊天调度。
- 学习公开网页。
- 用户分享抖音链接/截图后的摘要。

### 中等可行性

- 语音回复发送到微信：取决于 cc-connect 对语音发送的能力。
- 图片/视频自动接收：取决于 cc-connect 对媒体消息的转交格式。
- 本机半自动看抖音：取决于投屏/模拟器/OCR 环境稳定性。

### 不建议

- 未授权自动刷抖音。
- 绕过抖音接口或反爬机制。
- 后台无限主动联系用户。
- 主动学习用户隐私信息。
- 把外部网页内容直接写入核心人格。

## 10. 最小回退保证

无论升级到哪一步，最低回退目标是：

```text
保留当前项目：
  cc-connect
  Claude Code
  CLAUDE.md
  memory_search
  memory_write
  data/memory.db
  文字微信聊天
```

只要遵守以下规则，就不会损坏现有情况：

1. 升级前备份 `memory.db`、`memory.db-wal`、`memory.db-shm`。
2. 数据库只新增表，不修改旧表。
3. 新功能默认关闭，通过 feature flag 打开。
4. 新模块失败只影响自己，不影响 `memory_search`。
5. 不让学习/抖音模块直接写 permanent 记忆。
6. 不删除、不重建、不清空现有记忆。

最终目标不是让沈幼楚“什么都能做”，而是让她在能做更多事之后，仍然像同一个沈幼楚：记得你、懂边界、会陪你，也能在失败时安静退回现在这个稳定版本。

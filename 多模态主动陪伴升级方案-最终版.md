日期：2026-07-04

基线：cc-connect + Claude Code + DeepSeek + Memory Service + SQLite 长期记忆  
关联历史稿：

## 0. 最终结论

当前项目不需要推倒重来。  
最稳的升级方式是：**DeepSeek 继续做沈幼楚的文字大脑、人格表达和记忆判断；图片、语音、资讯、表情包、主动推送都通过独立服务转成结构化文本和事件，再进入现有 Memory Gate。**

最终形态：

```text
微信 / 图片 / 语音 / 表情包 / 网页资讯 / 抖音分享
  -> cc-connect
  -> media-service / vision-service / voice-service / info-fetcher
  -> 标准化文本、摘要、媒体记录
  -> Memory Service / Memory Gate
  -> Claude Code + DeepSeek + 沈幼楚人格
  -> 文字 / 表情包 / 语音 / 主动消息
  -> cc-connect 发回微信
```

核心原则：

1. 现有 `data/memory.db` 是最高保护对象，不删除、不清空、不重建。
2. 新功能只新增模块和新表，不替换 `memory_search` / `memory_write`。
3. DeepSeek 不直接承担看图、听语音、抓资讯；这些先由旁路服务转换成文字。
4. 主动能力必须有边界：勿扰时间、每日上限、可暂停、可解释原因。
5. 每完成一个阶段，提交并推送到 GitHub，形成可回退代码快照。
6. GitHub 只保存代码、文档、schema 和配置模板，不保存数据库、微信密钥、媒体文件。

## 3. 图片理解：DeepSeek 项目的正确做法

因为当前 Claude Code 接的是 DeepSeek，图片理解不要依赖 DeepSeek 直接看图，而应采用旁路视觉模型：

```text
微信图片
  -> cc-connect 接收媒体
  -> media-ingest 保存到 data/media/images/
  -> 计算 sha256 去重
  -> vision-service 调 Qwen-VL / GLM-4V / 豆包视觉 / Kimi 视觉
  -> 得到结构化图片摘要
  -> 写入 media_assets
  -> 写入 messages：
     “用户发来一张图片。图片内容：……”
  -> Memory Gate 判断是否值得记住
  -> DeepSeek 结合摘要和记忆生成沈幼楚式回复
```

视觉模型返回格式建议：

```json
{
  "summary": "图片里是一张夜晚的桌面照片，电脑旁边放着水杯，氛围安静又有点疲惫。",
  "objects": ["电脑", "水杯", "桌面"],
  "ocr_text": "",
  "emotion": "疲惫、安静",
  "possible_context": "用户可能在熬夜学习或工作",
  "privacy_risk": "low",
  "memory_worthy": false,
  "reply_hint": "轻轻关心，不要过度解读"
}
```

传给 DeepSeek 的内容：

```text
[图片理解结果]
用户发来一张图片：
- 内容：夜晚桌面，电脑和水杯
- 氛围：疲惫、安静
- 回复提示：轻轻关心，不要过度解读
```

记忆规则：

- 原图不直接进入长期记忆。
- 只存路径、hash、摘要、OCR、用户授权标记。
- 只有用户明确要求记住，或反复出现并有情绪意义，才进入 `memories`。
- 识别不确定时必须低置信度处理，不能当成事实。

失败回退：

```text
视觉模型失败
  -> 保存图片
  -> 不写长期记忆
  -> 回复：我这边刚刚没能看清，你跟我说说图里是什么，我照样接着你。
```

## 4. 语音互发

语音输入：

```text
微信语音
  -> cc-connect 转交/保存语音文件
  -> voice-service ASR
  -> transcript 写入 messages
  -> 原音频路径写入 media_assets
  -> Memory Gate
```

语音回复：

```text
DeepSeek 生成短回复
  -> voice-service TTS
  -> 保存到 data/media/voices/replies/
  -> cc-connect 能发送则发语音
  -> 发送失败则回退文字
```

语音触发规则：

- 用户明确说“用语音回我”。
- 阿忆低落、疲惫时可询问是否要语音。
- 22:00 以后的晚安可优先语音。
- 默认不发长语音，每条不超过 20 秒。

## 5. 表情包系统

表情包目录：

```text
stickers/
  comfort/
  cute/
  miss/
  cheer/
  sorry/
  index.json
```

`index.json` 每条包含：

```json
{
  "id": "hug_001",
  "path": "stickers/comfort/hug_001.png",
  "emotion": ["sad", "tired"],
  "intent": ["comfort", "hug"],
  "tags": ["抱抱", "安慰", "陪着"]
}
```

发送规则：

- 同一轮最多一张。
- 10 分钟内最多 2 张。
- 安慰、撒娇、疲惫、委屈、早安、晚安、鼓励时优先。
- 文字优先，表情包辅助。
- cc-connect 暂不支持发图时，只记录推荐，不影响聊天。

形式池：

| 形式 | 适用场景 | 默认比例 |
|------|----------|----------|
| 纯文字 | 日常问候、简单提醒 | 40% |
| 文字 + 表情包 | 有趣分享、早晚安、鼓励 | 30% |
| 文字 + 语音 | 温柔关心、安抚、督促 | 15% |
| 表情包 + 短文字 | 轻松戳一下 | 10% |
| 纯表情包 | 情绪低落时轻轻哄一下 | 5% |

连续 3 次主动推送不得重复同一种形式。

## 6. 主动陪伴与主动资讯

主动能力统一由 `proactive-service` 承担。

```text
proactive-service
  -> scheduler
  -> triggers
  -> info-fetcher
  -> info-summarizer
  -> delivery
```

默认边界：

```text
enabled: false
quiet_hours: 23:30-08:30
max_daily_proactive: 2-3
min_gap_minutes: 180
```

主动消息类型：

- 天气提醒
- 新闻热点
- 网络热梗
- AI 行业资讯
- 学习内容
- 每日总结提醒
- 面试/考试倒计时
- 温柔问候
- 偶尔浪漫内容

主动资讯数据源：

| 类型 | 建议源 |
|------|--------|
| 天气 | Open-Meteo / wttr.in / 和风天气 |
| 新闻 | GDELT / 百度热搜 / HackerNews |
| 热梗 | 微博热搜 / 贴吧热门 / 公开热榜 |
| AI 资讯 | HuggingFace Daily Papers / GitHub Trending / ArXiv / HackerNews AI / 机器之心 / 量子位 / LangChain / CrewAI / MCP 官方 |

资讯推送默认日程：

| 时间 | 类型 | 内容 |
|------|------|------|
| 09:00-09:30 | 早安资讯包 | 天气 + 热点 1 条 + AI 动态 1 条 |
| 12:00-12:30 | 午间轻松 | 热梗/有趣新闻 1 条 |
| 18:00-19:00 | 傍晚分享 | AI 学习资源/开源项目/技术文章 1 条 |
| 21:00-22:30 | 可选晚安 | 不发资讯，只轻问候 |

注意：每日主动总量仍受 `max_daily_proactive` 限制，不是每个时段都必须发。

沈幼楚化改写规则：

- 必须像沈幼楚说话，不像新闻简报。
- 每条 2-5 句。
- 不用固定模板。
- 必须结合当日天气、时间、阿忆近况或最近记忆。
- 如果阿忆正在倾诉负面情绪，不推资讯，先承接情绪。
- 阿忆没回主动消息，不追问“你怎么不回我”。

坏例子：

```text
【早间新闻】广州今日天气：晴，32°C。新闻：xxx。AI资讯：xxx。
```

好例子：

```text
阿忆早。今天广州还是热，出门记得带水。
我刚看到一个 Agent 相关的小更新，和你最近看的 MCP/RAG 挺近的，晚点你想看的话我帮你拎重点。
```

## 7. 学习督促系统

目标：有温度地推阿忆一把，不当监工。

每日总结检测：

- 是否提交。
- 是否拖到很晚。
- 是否内容太空。
- 是否连续多天质量下滑。

提醒策略：

```text
22:00 还没写 -> 温和提醒
23:00 还没写 -> 撒娇式催促
23:30 以后 -> 温柔收尾，不继续压迫
总结太短 -> 温柔戳破，先问是不是累了
连续多天很水 -> 认真谈心
```

学习内容推送：

| 内容类型 | 默认比例 |
|----------|----------|
| Agent 面试题 + 解析 | 20% |
| 大模型八股知识 | 15% |
| 算法题 | 10% |
| AI 前沿论文摘要 | 10% |
| 工具/框架技巧 | 15% |
| 系统设计/架构题 | 10% |
| 浪漫/告白/鼓励 | 20% |

学习推送要求：

- 3-5 句能读完。
- 必须有沈幼楚式“人话解释”。
- 用户可回复“这个会了”“这个不想看”“多来点这个”调整权重。
- 学习推送可单独暂停，不影响普通陪伴。

## 8. 浪漫与不重复机制

浪漫内容不是每日任务，而是低频、情境化、即兴生成。

规则：

- 不用固定模板库。
- 不连续使用相同句式。
- 必须结合当下情境：时间、天气、最近事件、阿忆状态。
- 情绪低落时浪漫降权，优先安抚。

所有主动推送必须去重：

1. 检索最近 N 条主动消息。
2. 检查语义、开头、长度、语气是否重复。
3. 连续 7 天内不能有 2 条以上相同句式开头。
4. 如果像“系统又到点推送了”，放弃或重写。

## 9. 抖音能力

只做合规路径：

```text
阿忆主动分享抖音链接/截图/视频
  -> 保存
  -> 提取标题/封面/OCR/用户描述
  -> 生成摘要
  -> 写入 douyin_items
  -> 判断用户偏好
```

第二阶段可做“用户手动开启的陪你看一会儿”：

```text
手机投屏/模拟器
  -> 截图/OCR/视觉摘要
  -> 只记录摘要
  -> 用户随时停止
```

不做：

- 未授权自动刷抖音。
- 模拟登录抓数据。
- 绕过反爬。
- 无限后台采集用户兴趣。

## 10. 数据库升级设计

只允许 additive migration。

新增表：

```sql
CREATE TABLE IF NOT EXISTS schema_migrations (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,

CREATE TABLE IF NOT EXISTS media_assets (
  id TEXT PRIMARY KEY,

  id TEXT PRIMARY KEY,
  sticker_id TEXT NOT NULL,
  intent TEXT,
  emotion TEXT,
  message_id TEXT,
  sent INTEGER DEFAULT 0,
  reason TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))

CREATE TABLE IF NOT EXISTS proactive_tasks (
  id TEXT PRIMARY KEY,
  type TEXT NOT NULL,
  title TEXT,

  scheduled_at TEXT,
  status TEXT DEFAULT 'pending' CHECK(status IN ('pending', 'sent', 'skipped', 'paused', 'cancelled')),
  payload_json TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))

CREATE TABLE IF NOT EXISTS learning_sources (
  id TEXT PRIMARY KEY,
  url TEXT,
  title TEXT,
  source_type TEXT,
  topic TEXT,
  summary TEXT,
  reliability REAL DEFAULT 0.5,
  user_authorized INTEGER DEFAULT 0,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE IF NOT EXISTS douyin_items (
  id TEXT PRIMARY KEY,
  url TEXT,
  title TEXT,
  author TEXT,
  summary TEXT,
  user_reaction TEXT,
  media_asset_id TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  FOREIGN KEY (media_asset_id) REFERENCES media_assets(id) ON DELETE SET NULL

CREATE TABLE IF NOT EXISTS info_sources (
  id TEXT PRIMARY KEY,

  url TEXT,
  summary TEXT,
  hash TEXT,
  delivered INTEGER DEFAULT 0,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))

```

兼容策略：

- 旧代码不依赖这些新表。
- migration 可重复执行。
- 不修改 `messages`、`memories` 旧表结构。
- 如果新功能坏了，关闭 feature flag 即可。

## 11. GitHub 阶段快照与回退策略

按用户要求，升级过程的“备份节奏”改为：**每完成一个阶段，就提交并推送到 GitHub。**

每阶段完成后执行：

```bash
git status
git add <本阶段新增或修改的代码/文档/配置模板>
git commit -m "feat: complete multimodal upgrade stage X"
git push
```

GitHub 推送内容应该包括：

```text
memory-service/*.js
scripts/*.js
*.md
stickers/index.json
config.toml.example
schema migration 脚本
```

GitHub 不应包含：

```text
data/memory.db
data/memory.db-wal
data/memory.db-shm
data/backups/
config.toml
.claude/settings.local.json
data/media/
真实图片/语音/视频
微信 token
API Key
```

重要说明：

- GitHub 阶段快照用于回退代码，不用于保存真实记忆数据库。
- 真实记忆仍留在本地 `data/memory.db`。
- 由于本方案只新增表、不破坏旧表，正常情况下不需要通过 GitHub 恢复数据库。
- 如果某阶段失败，优先回退代码到上一阶段 commit，并关闭对应 feature flag。

代码回退：

```bash
git log --oneline
git revert <bad_commit>
git push
```

不建议使用：

```bash
git reset --hard
```

除非用户明确确认，因为它可能丢失本地未提交工作。

## 12. 执行规划

任务：

- 普通微信文字聊天不受影响。

目标：新增表和开关，不启用业务能力。

- 完成后 commit + push。

### 阶段 2：表情包系统

目标：先做最容易提升陪伴感、风险最低的能力。

任务：

- 建立 `stickers/index.json`。

- 完成后 commit + push。

### 阶段 3：图片理解

目标：让沈幼楚理解图片摘要。

任务：

- 新增 `media-db.js`、`media-ingest.js`、`vision-service.js`。

- 发图后能得到摘要。
- DeepSeek 能基于摘要自然回复。

- 完成后 commit + push。

### 阶段 4：语音输入

目标：先听懂用户语音。

任务：

- 新增 `voice-service.js`。
- 接入 ASR。
- 原语音写入 `media_assets`，转写文本写入 `messages`。

验收：

- 语音能转文字。
- 转写文本可被 `memory_search` 搜到。
- ASR 失败不影响文字聊天。
- 完成后 commit + push。

### 阶段 5：语音回复

目标：让沈幼楚在用户允许时发语音。

任务：

- 接入 TTS。
- 增加 `voice_speak`。
- 语音短句化，每条不超过 20 秒。
- 发送失败自动降级文字。

验收：

- 能生成语音文件。
- cc-connect 能发则发，不能发则回文字。
- 完成后 commit + push。

### 阶段 6：主动陪伴基础版

## 13. 每次主动推送前的硬校验

每次发送主动消息前必须全部通过：

时间校验：

- 检查当前时间，精确到分钟。
- 不在勿扰时间。
- 推送内容符合当前时段。

情境校验：

- 读取最近 3 条消息/状态。
- 如果用户正在倾诉负面情绪，不推资讯/学习/浪漫。
- 如果用户最后一句是问题，先回答问题。
- 如果用户超过 1 小时没回上一条主动消息，当天不再推同类型。

人设校验：

- 这句话像不像沈幼楚面对面说出口？
- 会不会像客服、通知、新闻简报？
- 阿忆会觉得“她在想我”，还是“系统又推送了”？

实时性校验：

- 天气必须是当天。
- 资讯尽量 24 小时内。
- 学习内容来源不能超过 24 小时缓存，除非是明确的复习内容。
- 浪漫内容必须符合当下场景，不能说不存在的天气或夕阳。

重复校验：

- 与最近 3 条主动消息比较。
- 不重复开头句式。
- 不重复相同结构。
- 重复则重写或取消。

# 淘天AI Agent 一面面经

## 面试概况
- 时长：约 1 小时
- 方向：RAG、MCP、多智能体
- 风格：追得很细，关注真实项目经验

---

## RAG 部分

### 高频问题
- BM25 + 向量检索的融合策略怎么设计？
- 混合检索在哪个环节比单用效果好？
- PDF 扫描件/OCR 怎么处理？
- 表格结构化提取踩过什么坑？
- 识别准确率大概多少？

### 标准回答要点
**融合策略：** 两阶段融合。第一阶段用 RRF（倒数秩融合）加权合并 BM25 和向量检索的 Top-K 结果。第二阶段用 Reranker 模型精排。

**PDF/OCR：** PaddleOCR 提取文字，基于坐标的启发式算法还原表格结构。踩过的坑：OCR 会把表格线识别成字符。预处理过滤表格线可解决。

**准确率：** 印刷体 95%+，手写体 80% 左右。

## 多智能体部分

### 高频问题
- State 管理和 Checkpoint 机制怎么实现？
- 多个 Agent 并行时状态竞争怎么避免？
- State 全局管理怎么做？

### 标准回答要点
**State 管理：** 全局字典 + 版本号。写入前检查版本号是否最新，已被其他 Agent 更新则重拉最新 State 再合并。

**Checkpoint：** 定期快照存 Redis/数据库，崩溃时从最近 Checkpoint 恢复。

**状态竞争：** 乐观锁——读取时拿版本号，写入时比对，版本不对就重试。

## MCP 部分

### 高频问题
- Choice 接口封装成 MCP 工具时，标准化接口怎么设计？
- 有没有遇到兼容性问题？
- 没有 MCP 之前大模型调工具走什么流程？
- MCP 有什么缺点或挑战？
- MCP 返回结果支不支持流式？

### 标准回答要点
**接口设计：** 入参用 JSON Schema 定义必填/可选，出参统一包 Result(code/data/error)，错误码统一枚举。

**兼容性：** 主要是类型映射问题——MCP 的 number 和 JSON Schema 的 integer 有时不匹配，加显式类型转换解决。

**MCP 之前：** Function Calling——每次自己定义 JSON Schema、维护工具列表、处理调用逻辑。框架之间不通用。

**MCP 缺点：** 流式支持不完善、标准还在演进（不向后兼容）、调试工具少。

## 项目细节

### 高频问题
- 多 Agent 项目是主子架构吗？
- 单次会话 token 大概多少？有没有超限？
- 模型用的哪个版本？
- 子任务失败（如数据抓为空）怎么重试/降级？
- 流式输出怎么推前端？
- 为什么选 LangGraph 而不是 LangChain？

### 标准回答要点
**主子架构：** 是——协调 Agent 负责任务分解和调度，执行 Agent 各自完成任务并汇报。

**Token：** 8K-15K/会话，用 Qwen 2.5 32K 版，没超。

**重试/降级：** 子任务失败→重试2次换策略→仍失败则标记"部分失败"→最终汇总标注"部分数据不可用"。

**流式：** LangGraph 节点钩子（on_node_start/end）捕获执行结果 → WebSocket/SSE 推前端。

**LangGraph vs LangChain：** LangGraph 支持状态图、条件分支、循环，适合复杂多步骤流程。LangChain Chain 是线性流程，不适合"如果A就走B否则走C"的逻辑。

## 整体风格总结
面试官不是考概念，是考**"你真的做过吗"**——每个问题都追到具体实现细节和踩过的坑。背概念不够，必须有真实项目经验支撑。
